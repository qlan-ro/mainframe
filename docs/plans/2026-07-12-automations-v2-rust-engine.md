# Automations v2 Rust Engine — Implementation Plan

**Goal:** A greenfield Automations v2 engine (When-triggers + linear Do-steps, four verbs, two blocks, token chips) built as a new crate **inside the existing `packages/core-rs` Rust daemon**, mounted into `mainframe-server` and wired through `mainframe-chat`/`PushService`/the broadcast bus at daemon boot — closing the CUTOVER "Workflows engine deferred" gap on the Rust arm.

**Architecture:** One new library crate `mainframe-automations` (domain, store, interpreter, scheduler, actions) whose every external effect is a **trait port** with a testkit fake — the same `ChatManagerDeps`/`DaemonChatDeps` inversion the daemon already uses, so the engine crate never depends back on `mainframe-server` (no cycle). Production port impls (`DaemonAutomationsDeps`) and the axum route modules live in `mainframe-server`; boot construction lives in `mainframe-daemon`. Durability follows v1's replay model over one `checkpoint` JSON column: `advance()` re-walks the definition, skips committed steps, executes the first live step, commits, repeats. Scheduler state is derived from the runs table; a 30 s sweep drives it (no `sleep` timers — laptops sleep).

**Tech stack (core-rs `[workspace.dependencies]`, matched):** edition 2024, resolver 2, tokio 1, axum 0.8, **rusqlite 0.32 (`bundled`)**, serde/serde_json, chrono 0.4, `hmac`/`sha2`/`hex`, reqwest 0.12 (rustls), `nanoid`, `dashmap`, thiserror 2, tracing; new workspace deps to add: **croner 3** (cron), **rmcp 2** (MCP client), dev-only **wiremock 0.6** + existing `tempfile`.

**Ground-truth sources:** `docs/designs/2026-07-11-automations-v2-spec.md` (§12 reference automations); **ratified contract `docs/plans/2026-07-12-automations-v2-contract.md`** (wire shapes/routes/events/schema/A1–A3 — FROZEN, do not diverge); `docs/rust-port/{CUTOVER,CANARY-REPORT,PORTING}.md`; v1 semantics reference only: `packages/core/src/workflows/`.

## Global constraints
- **Retargeted:** no `packages/daemon-rs/`. All new code lands under `packages/core-rs/`. No standalone binary — the engine ships in-daemon.
- **core-rs Rust rules (`PORTING.md` §4, enforced by `tools/verify-gate.sh`):** `#![forbid(unsafe_code)]`; **no `anyhow` in the library crate** (thiserror error enums); no `unwrap()`/`expect()`/`panic!()`/`todo!()`/`unimplemented!()` outside `#[cfg(test)]`; tokio only; serde `#[serde(rename_all="camelCase")]` + `skip_serializing_if="Option::is_none"` on optionals unless a fixture shows explicit `null`; ≤300 lines/file. Gates from `packages/core-rs/`: `cargo fmt --check`, `cargo clippy --all-targets -- -D warnings`, `cargo test`, `tools/verify-gate.sh`.
- **CI is already wired:** `.github/workflows/rust-port.yml` is path-filtered to `packages/core-rs/**`, so the new crate + `mainframe-server`/`mainframe-types`/`mainframe-daemon` edits run the four gates automatically. No new workflow. Rust-only PRs still need `pnpm changeset --empty`.
- **DB:** the engine owns a **separate `<dataDir>/automations.db`** (WAL, `busy_timeout=5000`, `foreign_keys=ON`) — the three contract tables only. It is **not** part of `mainframe.db`'s migration chain (`LATEST_VERSION=24`, untouched), preserving the CUTOVER lock-step-migration invariant. `dataDir` = `MAINFRAME_DATA_DIR|~/.mainframe`, via `dirs`.
- Contract fixtures live at `packages/types/fixtures/automations/*.json`; tests load them by relative path. They must exercise A1–A3.

## Locked decisions
- **Crate: repurpose the empty `mainframe-workflows` placeholder → rename to `mainframe-automations`.** That crate is a stub (`lib.rs` comment "empty placeholder … real port deferred", 4 deps) and **nothing depends on it** (verified). The v1 workflows port was abandoned by scope decision, and the product renamed the feature to "Automations", so a v1-named empty crate would mislead. Rename the dir + its `[workspace.dependencies]` path entry; `members=["crates/*"]` needs no change.
- **Ports are traits; production impls live in `mainframe-server`.** The engine crate defines `AgentPort`, `Notifier`, `EventSink`, `ProjectRegistry`, `McpConfigSource`, `CredentialStore`, `Clock` and a `testkit` feature with a fake for each. `mainframe-server::automations_deps::DaemonAutomationsDeps` implements them over live handles; `build_automations_engine(...)` (mirroring `build_chat_manager`, `chat_deps.rs:510`) constructs the engine. `mainframe-automations` depends only on `mainframe-types` (+ leaf deps) — never on `mainframe-server`/`-chat`/`-db`.
- **Checkpoint = one JSON doc** (`automation_runs.checkpoint`, contract §2): `{ "steps": { "<stepRef>": StepEntry }, "wakeAt": number|null, "error": string|null }`. `stepRef` = stepId top-level/If-branch, `<stepId>#<iter>` inside Repeat, chained for nesting. `StepEntry = { stepId, kind, status: succeeded|failed|waiting|skipped, outputs: Record<String,Value>|null, error, startedAt, finishedAt?, chatId?, interactionId? }`. Outputs are **named** (indexed by `TokenRef.output`), stored whole; per-step cap 4 MB, failing loudly with the "write large data to a file" message. Timeline API projects entries with 32 KB display-truncation; no step_runs table.
- **Scheduler state derived, not stored.** Each fire records `scheduledFor` in the run's `trigger` JSON. Sweep computes the latest occurrence ≤ now; if no run exists for `(automationId, triggerId, scheduledFor)` → fire when fresh (< 5 min late), else `onMissed`: `run_once` fires one make-up, `skip` none. Survives restart/sleep with zero extra tables.
- **Credentials: 0600 JSON file** `<dataDir>/automation-credentials.json` (v1 pattern) — core-rs has **no reusable secret store** (only `ensure_auth_secret`, verified), so the engine owns this. Webhook signing secrets stored here under reserved labels `webhook:<hookId>`, never in the exportable definition.
- **Statuses (contract §1).** Runs `running|waiting|succeeded|failed|cancelled`; interactions `pending|answered|cancelled` (`cancelled` on run-cancel; interactions never expire).
- **Restart mid-action:** a `running` entry on resume re-executes if the action manifest is `idempotent`, else the step fails "engine restarted mid-action; effect unknown" (keep-going flag applies). v1's `ambiguous` collapses into `failed`.

## Cutover fit (per `docs/rust-port/CUTOVER.md`)
This engine lands behind the existing `MAINFRAME_DAEMON_IMPL` flag; it does **not** flip the default. It closes the §3 "Workflows engine (deferred)" gap and the §4 "Workflows unavailable on the Rust daemon" release-note item **on the Rust arm only**, once green. `mainframe.db` migrations stay lock-step (the engine's `automations.db` is a separate file — CUTOVER §2 caveat unaffected). Parity: because the Node plan implements the same ratified contract, the new `/api/automations*` routes join the CUTOVER step-3 `diffd` route set against Node once both surfaces exist; the shared fixtures are the golden.

## Risks / open questions
- **R1 Wire-shape ratification — RESOLVED** by the contract + `packages/types/fixtures/automations/*.json`.
- **R2 Shared-DB switchover — owned by the cutover runbook.** `automations.db` is a separate file outside `mainframe.db`'s chain, so node↔rust flips (CUTOVER §2) leave it untouched. If a future build lets both daemons run automations concurrently, the runbook must add a single-writer/ownership rule (WAL makes overlap safe but double-scheduling undefined). No new tables; no migration-25 needed on `mainframe.db`.
- **R3 Webhook sample capture — RESOLVED.** In-memory in the daemon; editor exposure is a UI-plan route, not yet wired.
- **R4 Notification delivery — RESOLVED against core-rs.** Desktop = the WS `automation.notification` event the Tauri shell renders (same path as `chat.notification`); mobile = `PushService::send_push`. Both are real, wired at boot — no future-plan dependency.
