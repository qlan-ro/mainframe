# Automations v2 Rust Engine â€” Implementation Plan

**Goal:** A greenfield Rust implementation of the Automations v2 engine (When-triggers + linear Do-steps, four verbs, two blocks, token chips) that runs today as a standalone axum process against the shared SQLite contract, and later slots into the Rust daemon unchanged.

**Architecture:** A new Cargo workspace `packages/daemon-rs/` with three crates: `automations` (domain, store, interpreter, scheduler, actions â€” all external effects behind traits), `automations-http` (axum + WS surface, WS4 envelope), `automations-node-agent` (optional Ask-agent adapter driving the Node daemon's chat API). Durability follows v1's replay model, compressed into the contract's single `checkpoint` JSON column: advance() re-walks the definition, skips committed step entries, executes the first live step, commits, repeats. No timers via sleep â€” a 30 s sweep drives schedules; scheduler state is derived from the runs table, not a private table.

**Tech stack:** Rust stable (pin 1.85 in `rust-toolchain.toml`), tokio 1, axum 0.8.9, rusqlite 0.40 (`bundled`), serde/serde_json, chrono 0.4 + croner 3.0, reqwest 0.12 (rustls), rmcp 2.2, hmac/sha2, wiremock 0.6 + tempfile (dev), thiserror, tracing.

**Spec sources (binding):** `docs/designs/2026-07-11-automations-v2-spec.md` (model, Â§12 reference automations); the shared REST/WS/SQLite/semantics contract fixed in the task brief; v1 semantics reference only: `docs/plans/2026-06-12-workflow-engine-v1.md`, `packages/core/src/workflows/`.

## Locked decisions
- **rusqlite over sqlx.** Same-file sharing with better-sqlite3 wants plain synchronous SQLite with WAL + busy_timeout; checkpoint writes are tiny transactions. sqlx's async adds nothing for a local file and its compile-time checking needs CI DATABASE_URL plumbing. All store calls go through `tokio::task::spawn_blocking`; one writer connection behind a `parking_lot::Mutex`.
- **Checkpoint = one JSON doc** (`automation_runs.checkpoint`, canonical shape from contract Â§2): `{ "steps": { "<stepRef>": StepEntry }, "wakeAt": number|null, "error": string|null }`. `stepRef` = stepId top-level/If-branch, `<stepId>#<iter>` inside Repeat, suffixes chained for nesting. `StepEntry = { stepId, kind, status: succeeded|failed|waiting|skipped, outputs: Record<String,Value>|null, error, startedAt, finishedAt?, chatId?, interactionId? }`. Outputs are **named** (keyed by output name â€” `TokenRef.output` indexes them) and stored whole; per-step hard cap 4 MB, failing loudly with the "write large data to a file" message. The timeline API projects entries (32 KB display-truncation); there is no step_runs table.
- **Scheduler state derived, not stored.** Each schedule fire records `scheduledFor` inside the run's `trigger` JSON. Sweep: for each enabled schedule trigger, compute the latest occurrence â‰¤ now; if no run exists for (automationId, triggerId, scheduledFor) â†’ fire when fresh (< 5 min late), else apply `onMissed`: `run_once` fires one make-up (with the missed `scheduledFor`), `skip` does nothing (the watermark is implicit â€” the next occurrence computes forward from now). Survives restarts and sleep with zero extra tables. Node's engine-internal `trigger_state`/`agent_waits` caches are not contract; this plan neither reads nor mirrors them, and tolerates unknown tables in the shared file.
- **Credentials: 0600 JSON file** at `<dataDir>/automation-credentials.json`, v1 pattern (`packages/core/src/workflows/credentials.ts`). A background engine process can't reliably use the macOS keychain without per-item prompts, and the Node engine must read the same store. `CredentialStore` is a trait; keychain is a future impl. Value shapes: `{kind:"bearer",token} | {kind:"header",name,value} | {kind:"basic",user,pass}`. Webhook signing secrets live here under reserved labels `webhook:<hookId>` â€” never in the definition JSON (it's exportable).
- **Statuses (contract Â§1).** Runs: `running|waiting|succeeded|failed|cancelled`. Interactions: `pending|answered|cancelled` (`cancelled` set on run-cancel; interactions never expire).
- **Restart mid-action:** a `running` entry found on resume re-executes if the action manifest says `idempotent`, else the step fails with "engine restarted mid-action; effect unknown" (keep-going flag applies). v1's `ambiguous` status is collapsed into `failed`.
