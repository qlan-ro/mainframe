## Rust-side fixes applied during this run

The first run surfaced two Rust divergences (now fixed; totals above are post-fix):

1. **`continue-in-project-root` / `continue-here` never persisted (DIVERGENT).**
   `chat[1].branch_name` stayed `"diffd-ghost-branch"` on the Rust side after
   `continue-in-project-root` returned `200 okEmpty`, but was `NULL` on Node. Cause:
   `ChatManagerDeps::chats_clear_session` / `chats_clear_worktree` were **empty
   default trait methods**, and the production `DaemonChatDeps`
   (`mainframe-server::chat_deps`) never overrode them — so both recovery ops were
   silent no-ops against the DB while still emitting `okEmpty`. Fix:
   `DaemonChatDeps` now implements both (bridging to `db.chats.clear_session` /
   `clear_worktree`, mirroring `chats_update`), and the two trait methods were made
   **required** (no default) so this footgun cannot recur — the TS `ChatManager`
   always holds the non-optional `db.chats.clearSession`/`clearWorktree`.
2. **`mainframe-daemon` did not build (E0063).** `plugin_host_db.rs`'s defensive
   create-failure `Chat { .. }` stub was missing the Cluster-A fields
   (`lastContextTotalTokens`, `lastContextMaxTokens`, `transcriptMissing`,
   `backgroundActivity`); added all four as `None`.

## Scope notes

- Chat.backgroundActivity is derived from the in-memory live-task tracker (BackgroundTaskTracker.listLive), which is populated only by (a) a live CLI session emitting task events or (b) boot reconcile of a bash spool whose file a live writer process still holds (lsof). Neither has a deterministic HTTP path, so this harness verifies the derived-absent parity (no live task ⇒ field omitted on both — see chat-context-usage-fields and chat-get) and defers the populated-activity shape to the ported unit tests (chat-manager-background-activity, tracker) and the live soak (soak.mjs).
- The recovery routes are driven against a SQL-injected dead-worktree chat (chat[1]) and the enriched-Chat probe against a SQL-injected context-usage snapshot (chat[2]) because no deterministic HTTP mutation writes those columns — they are set by the CLI event handler (onContextUsage) / transcript reconcile at runtime.
